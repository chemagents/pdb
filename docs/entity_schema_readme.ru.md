# Руководство По Схемам Сущностей

Эти файлы схем нужны для того, чтобы человек и агентные системы разделяли строгое и повторно используемое понимание чувствительных объектов реального мира, используемых в моделировании. Они также служат заготовками для онтологии предметной области.

## Назначение

Используй эти схемы, когда агенту нужно:

- извлекать структурированные данные из реестров вроде PDB, UniProt, AlphaFold, PubChem и связанных источников
- нормализовать записи между несколькими структурными и химическими реестрами
- сохранять единый смысл полей для структур рецепторов, комплексов с лигандами и самих лигандов
- сохранять единый смысл полей для структур рецепторов, сайтов связывания, комплексов с лигандами и самих лигандов
- не делать скрытых догадок в неоднозначных случаях
- строить согласованные с онтологией наборы данных для докинга, молекулярного моделирования, аннотации и интеграции знаний

## Правила Чтения Для Агентов

Когда агент читает файл схемы, он должен воспринимать его как семантический контракт.

Агент должен:

1. определить тип сущности и гранулярность записи до извлечения значений
2. сохранять названия полей в точности как в схеме
3. соблюдать допустимые значения там, где заданы перечисления
4. использовать `null`, если надежные данные отсутствуют
5. не выводить значения, которые не подтверждаются доверенными источниками
6. сохранять происхождение данных через поля вроде `structure_registry`, `structure_id`, `external_ids` и `registry_ids`
7. оформлять связи между сущностями через `related_entities`
8. хранить одну запись на одну единицу гранулярности, указанную в схеме

## Правила Нормализации

- Нельзя считать PDB единственным источником структурной истины.
- Нельзя сливать разные химические состояния, если схема считает их отдельными значимыми сущностями.
- Нельзя считать буферные компоненты, соли, молекулы растворителя и криопротекторы значимыми лигандами, если источник или куратор явно не указал обратное.
- Если источник различает биологически релевантную сборку и просто депонированную геометрию, нужно предпочитать биологически релевантную интерпретацию.
- `receptor_structure`, `binding_site`, `ligand_complex` и `ligand` должны оставаться отдельными сущностями, даже если они тесно связаны.

Практическое замечание: текущая эвристическая логика разделения `apo`/`holo` и выделения primary ligand code может быть неполной. Если в записи есть несколько nonpolymer components, агент должен консервативно отличать ключевой функциональный лиганд от сопутствующих additives, ions, glycans и других нефункциональных компонентов.

Для `receptor_structure` поле `bound_ligands` нужно воспринимать как отфильтрованный список meaningful ligand component codes, а `primary_ligand_code` как более строгое поле: его следует заполнять только тогда, когда один функционально центральный ligand можно выбрать консервативно. Если meaningful ligands несколько или ранжирование неясно, `primary_ligand_code` должен оставаться null, а ambiguity нужно явно описывать в `notes`.

Для `binding_site` поле `residue_members` нужно воспринимать как консервативный набор остатков сайта, а `active_residue_members` как более строгий поднабор. Нельзя автоматически считать все ligand-contacting residues активными остатками. Если источник поддерживает только ligand-contact или software-generated binding-site evidence, интерпретация сайта должна оставаться консервативной, без догадок о catalytic роли.

## Рекомендуемый Рабочий Процесс Для Агентов

1. выбрать схему целевой сущности
2. прочитать `description`, `entity_definition` и `record_granularity`
3. просмотреть `fields` и составить чеклист извлечения по полям
4. прочитать `interpretation_rules` до разбора неоднозначных случаев
5. формировать результат в соответствии с `output_template`
6. сохранять нерешенную неоднозначность в `notes`
7. связывать объекты через `related_entities`

## Минимальная Дисциплина Вывода

- сохранять точные имена полей
- сохранять стабильные типы
- сохранять стабильные перечисления
- явно указывать происхождение данных
- явно показывать неопределенность

## Использование В Онтологии

Эти схемы подходят как практические заготовки для онтологии. Они задают:

- границы сущностей
- семантику полей
- связи между сущностями
- ожидания по нормализации
- паттерны связи с реестрами

В дальнейшем их можно перевести в JSON Schema, SHACL, OWL, RDF-отображения, модели БД или правила памяти для агентов.
